# domain/models.py - структура `Chunk`

Единственная доменная сущность: фрагмент кода (функция/метод/класс) с метаданными. Всё в системе оперирует `Chunk` или производными от него dict.

```python
from dataclasses import dataclass, field
```
- `dataclass` - автогенерит `__init__`, `__repr__`, `__eq__` по аннотированным полям: меньше шаблонного кода.
- `field` - нужен для поля со значением по умолчанию, которое является изменяемым объектом (список). Нельзя писать `calls: list = []` - общий список на все экземпляры; `field(default_factory=list)` создаёт новый список под каждый объект.

```python
@dataclass
class Chunk:
    chunk_id: str          # ИД в формате scorer: "{relpath}:{name}:{line}"
    source: str            # имя набора/репозитория - для админки и фильтрации
    lang: str              # "python"|... - какой парсер создал; для подсветки в UI
    file: str              # относительный путь от корня репо
    type: str              # function | class | method
    name: str              # имя символа (без класса)
    parent: str | None     # имя класса для метода, иначе None
    start_line: int        # строка def/class (node.lineno)
    end_line: int          # конец тела (node.end_lineno)
    code: str              # исходный текст фрагмента (хранится в сторе → показываем без файлов)
    docstring: str | None  # докстринг, если есть
    calls: list[str] = field(default_factory=list)  # имена вызываемых функций
```
- Поля до `calls` обязательны (нет дефолта) - конструктор требует их явно.
- `parent` отделено от `name`, потому что в `chunk_id` они склеиваются как `Class.method`, но в UI и метаданных удобно иметь их по отдельности.
- `code` хранится прямо в чанке (а потом в сторе) - это позволяет показывать результат и работать без исходных файлов после индексации.

## Метод `enriched_text` - что реально идёт в эмбеддер
```python
def enriched_text(self) -> str:
    head = (f"Метод {self.name} класса {self.parent}." if self.parent
            else f"{'Класс' if self.type == 'class' else 'Функция'} {self.name}.")
```
- Построение «шапки» на естественном языке. Тернарник: если есть `parent` → это метод, описывается «Метод X класса Y»; иначе по `type` выбирается «Класс»/«Функция». NL применяется потому, что запросы пользователей - фразы вроде «как создаётся токен», и они ближе к описанию, чем к синтаксису кода; эмбеддинг описания лучше матчится с запросом, особенно RU-запрос против EN-кода.

```python
    parts = [head]
    if self.docstring:
        parts.append(self.docstring)
```
- Докстринг - самый ценный сигнал (человеческое описание назначения); добавляется, если есть.

```python
    parts.append(self.code.splitlines()[0] if self.code else "")
```
- Первая строка кода = сигнатура (`def create_access_token(data, expires):`). Несёт имена аргументов/типов - полезно для совпадений. Защита от пустого `code` (тернарник), чтобы `splitlines()[0]` не упал на `IndexError`.

```python
    if self.calls:
        parts.append("Вызывает: " + ", ".join(self.calls[:10]))
    return "\n".join(parts)
```
- Список вызываемых функций добавляет контекст связей («функция, которая вызывает `jwt.encode`»). Обрезается до 10, чтобы не раздувать текст шумом. Части склеиваются через перевод строки → один текст для эмбеддинга.

`enriched_text` размещён на `Chunk`, а не в эмбеддере: обогащение - свойство данных, а не модели; так его проще переопределить (см. `enrich.py`) и протестировать без загрузки моделей.
