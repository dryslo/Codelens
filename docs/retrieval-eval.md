# Замер вклада retrieval-каналов (без реранкера)

Дата прогона: 2026-05-28. Корпус: 155 чанков (`data/codebase`, индекс в `.chroma`). Эвал: 15 вопросов из `data/eval_questions.json`. Эмбеддер: `intfloat/multilingual-e5-large`. Реранкер отключён по умолчанию ([config/config.yaml:12](../config/config.yaml#L12)). LLM для HyDE/MultiQuery: `llama-3.3-70b-versatile` на Groq.

Прогон через `evaluate.py` с переключателями [SearchFlags](../src/retrieval/flags.py), без реранкера во всех конфигах.

> Выбор эмбеддера. Как альтернативу проверяли `ai-forever/FRIDA` - на этом корпусе
> `intfloat/multilingual-e5-large` оказался лучше, поэтому он остаётся дефолтом
> ([config/config.yaml](../config/config.yaml)). FRIDA выдаёт 1536d (e5-large - 1024d), так что
> для возврата к FRIDA нужно задать `embedder.dim: 1536` и переиндексировать.

## По каждому каналу - почему так

### BM25 - 0 прироста (02, 07, 08, 10)

Лексический канал нашёл ровно те же чанки, что и dense. Чистый дубль.

Причина: на этом корпусе у dense уже Hit@5 = 93.3% (14/15 вопросов имеют правильный чанк в топ-5) - потолок recall'а достигнут до фьюжна. RRF может только пересортировать общие позиции, но не добавляет нового. Плюс `multilingual-e5-large` в 1024d уже схватывает лексический сигнал идентификаторов (`get_logger`, `Settings`, `root` - это OOV-токены, на которых эмбеддинги обычно слабые, но e5-large их вытягивает).

Когда BM25 начнёт давать прирост: на больших корпусах (тысячи+ чанков, где Hit@5 падает ниже потолка), с длинными «грязными» идентификаторами/аббревиатурами, или когда в коде много транслитераций/слов, которых нет в обучающей выборке эмбеддера.

### MMR - стабильный минус (03, 06, 11)

Диверсификация выкидывает релевантное: P@5 −8.9, Hit@5 −6.6.

Причина: в `eval_questions.json` для многих вопросов `correct_chunk_ids` содержит несколько похожих чанков (методы одного класса, перегрузки, родственные функции). MMR с λ=0.7 наказывает за похожесть на уже выбранное - и выкидывает именно эти «правильные дубли». В этом датасете оценивается покрытие связанных функций, а не разнообразие тем.

Когда MMR начнёт помогать: чат-режим с ограниченным контекстом (когда дубли просто съедают токены LLM), эвал с метрикой recall@k по разным темам, или когда индекс содержит много почти-идентичных версий одного и того же (форки, миграции).

### HyDE - −4.4 P@5 в одиночку, нейтрален с BM25 (04, 07, 10)

Один HyDE снижает precision (68.9% vs 73.3%), Hit@5 сохраняется → правильные чанки остаются в выдаче, но проседают в ранге.

Причина: LLM генерит «гипотетический фрагмент кода» и склеивает с запросом перед эмбеддингом. Для коротких функциональных запросов («где логгер настраивается?») гипотеза тянет эмбеддинг к среднему «канону Python-логгинга» - в результате чанки с похожим, но не точным кодом получают больше веса.

С BM25 эффект гасится: лексический канал фиксирует точные совпадения по именам идентификаторов и компенсирует размытие эмбеддинга гипотезой.

Стоимость: 6× времени (87s vs 14s) - sequential LLM call на каждый запрос.

Когда HyDE начнёт помогать: запросы естественным языком, далёкие от кода («что происходит при сбое подключения к БД?»), длинные описательные вопросы, кросс-язычные запросы (вопрос на RU - код на EN с малоговорящими именами).

### MultiQuery - 0 прироста, 15× замедление (05, 08, 10)

3 переформулировки + оригинал, каждая отдельная dense-выдача, RRF-фьюжн → ровно baseline.

Причина: переформулировки не открывают новых чанков. Оригинальный запрос уже находит всё, что есть. Variant query - это в основном синонимы и перестановки слов; для технических запросов про код синонимы редко что меняют (`get_logger` ≠ «функция для логов»).

Стоимость: 15× времени (220s) - 1 LLM-вызов + 4× dense-запросов.

Когда MultiQuery начнёт помогать: двусмысленные запросы, мультиязычность (юзер пишет на RU, в коде/документации EN), пользовательские формулировки сильно расходятся со словарём индекса.

### Комбинации LLM-каналов

- HyDE+MultiQuery (09): P@5 70%, Hit@5 86.7%, 480s. Хуже каждой в отдельности - два LLM-шума накладываются, теряется один правильный ответ. 34× времени.
- BM25+HyDE+MultiQuery (10): BM25 спасает recall (Hit@5 обратно 93.3%), но HyDE-размытие даёт −1.1 P@5. 28× времени за P@5 ниже baseline.
- All-no-rerank (11): MMR-минус доминирует, P@5 65.6%.

## Главный вывод

На малом чистом корпусе с эвалом, заточенным на «найти все связанные чанки», сложный пайплайн систематически проигрывает простому. Recall насыщен dense'ом, а каждая надстройка либо:

- избыточна (BM25, MultiQuery): не добавляет новых попаданий,
- противоречит метрике (MMR): убирает то, что эвал считает правильным,
- размывает ранжирование (HyDE): тянет к шаблонному коду.

Плюс стоимость LLM-каналов нелинейная: HyDE +6×, MultiQuery +15×, вместе +34× времени.

### Когда стоит пересматривать вывод

Этот замер валиден только для (тиражируемый Python-репо, 155 чанков, эвал с групповыми правильными ответами, e5-large эмбеддер). При изменении одного из трёх условий выводы меняются:

| Изменение контекста | Что включить заново |
|---|---|
| Корпус > 5k чанков | проверить bm25, mmr, hyde |
| Эвал/задача про разнообразие (recall@k по темам, chat-контекст) | mmr |
| Запросы на NL далёкие от кода | hyde |
| Мультиязычные/расплывчатые запросы | multiquery |
| Эмбеддер слабее e5-large | bm25 почти наверняка даст прирост |

Сетка переключателей в [SearchFlags](../src/retrieval/flags.py) и тумблеры в UI/CLI ([app.py](../app.py), [evaluate.py](../evaluate.py)) позволяют эти гипотезы проверять одной командой.

> Почему все добавочные каналы выключены по умолчанию. На small-корпусе они вредят
> или не дают прироста, поэтому в `config/config.yaml` → `retrieval.flags` все стоят
> в режиме `ui` (юзер сам решает) и выключены по умолчанию, а `rerank` принудительно
> `off`. На профиле large (>5k чанков, мультиязычные запросы) ожидается, что хотя бы
> `bm25` (как `fast`) и `hyde`/`multiquery` (как `thinking`) начнут давать прирост -
> это будет показано отдельным прогоном матрицы на large-корпусе из админки
> (вкладка Админка → Матричный прогон).

## Воспроизведение

```bash
for combo in \
  "01_dense:--no-bm25 --no-mmr --no-hyde --no-multiquery --no-rerank" \
  "02_bm25:--bm25 --no-mmr --no-hyde --no-multiquery --no-rerank" \
  "03_mmr:--no-bm25 --mmr --no-hyde --no-multiquery --no-rerank" \
  "04_hyde:--no-bm25 --no-mmr --hyde --no-multiquery --no-rerank" \
  "05_multiquery:--no-bm25 --no-mmr --no-hyde --multiquery --no-rerank" \
  "06_bm25_mmr:--bm25 --mmr --no-hyde --no-multiquery --no-rerank" \
  "07_bm25_hyde:--bm25 --no-mmr --hyde --no-multiquery --no-rerank" \
  "08_bm25_multiquery:--bm25 --no-mmr --no-hyde --multiquery --no-rerank" \
  "09_hyde_multiquery:--no-bm25 --no-mmr --hyde --multiquery --no-rerank" \
  "10_bm25_hyde_multiquery:--bm25 --no-mmr --hyde --multiquery --no-rerank" \
  "11_all_no_rerank:--bm25 --mmr --hyde --multiquery --no-rerank" \
; do
  label="${combo%%:*}"; args="${combo#*:}"
  start=$(date +%s.%N)
  out=$(python evaluate.py $args 2>/dev/null | grep "Precision@5")
  dt=$(awk "BEGIN{printf \"%.1f\", $(date +%s.%N)-$start}")
  echo "$label  |  $out  |  ${dt}s"
done
```
